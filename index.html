Nest Engine

2020 - Present

A complete game engine written from the ground up for the Nintendo 64 in C++17.

Info

Nest Engine was created after two months of development for a game jam competition. The Swoop 64 was thus created using the engine and won third place.

  • Streamed audio API. Samples themselves are VADPCM-encoded and are decoded into an audio heap, which is eventually DMA'd to the Audio DAC. The audio heap can be expanded to any size needed. Its location is at the very end of memory, right above the game's framebuffer. Of course, changing the heap location is allowed, but has to be approached with caution.
  • Versatile, intuitive vector and matrix API for complex graphical calculations.
  • 3D collision system for both actor/object and floor/ceiling interaction.
  • Primitive, yet effective animation system. The Nintendo 64 has a rough skinned mesh limitation, so one method around this was to implement a vertex animation system. The size of vertices raised concern, so each vertex is lerped between frames.
  • Relocatable module system for code and data that cannot fit within Nintendo 64 's four megabyte memory range. A tool was written to parse pre-compiled objects, extract the relocation table and data, and finally use this data to relocate the aforementioned code by overwriting it directly in memory during game runtime.
  • For creating/modifying/editing game data such as models, object positions, and etc., a custom Blender plug-in was written to export model and scene data for the Nintendo 64 to parse. N64 uses a graphics API that is very similar to legacy OpenGL, as it was created by SGI. Each graphics command gets compiled into a 64-bit command, forming a display list, which is then passed to the secondary CPU to process.

Technologies used:

  • C++
  • Blender
  • Audacity

Super Jolly Engine

2017 - Present

Custom game engine for the Super Nintendo Entertainment System written in 65C816 assembly.

Info

As no personal nor commercially released games have been developed with the engine yet, it is fully capable of running sprite-intensive and high performant game software!

  • Fully custom audio driver, called F700. Written in assembly and runs on the SPC700 architecture, which is an 8-bit CPU that channels data between it and the DSP chip. F700 is based on module trackers, so music and sounds are processed rows and columns of data. The audio CPU does not have interrupts, so time is based on three MMIO registers you poll while the processor runs. The driver polls the input registers from the main CPU until a command is received; i.e., the game engine sends a byte to the DSP telling it to play a sound at index X. Samples are 4-bit ADPCM-encoded.
  • Parent-child object system designed for easy object manipulation and management.
  • Versatile assembler written to support a wide range of instruction set architectures. Uses an instruction set table syntax file for parsing instructions on the fly, designed to read any kind of architecture desired.
  • Tool was written to take standard formatted images and convert them to the SNES' bitplane data format.

Technologies used:

  • 65C816 Assembly
  • SPC700 Assembly
  • C++
